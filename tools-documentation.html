1. Frontend Technologies

Frontend HTML5 (HyperText Markup Language)
HTML5 is the standard markup language used to build the structure of every page in the application. It defines what appears on screen using elements such as headings, paragraphs, forms, tables, buttons, and links. In Kigali Estates, HTML5 is used across all pages including index.html (homepage), properties.html (listings), cart.html (shopping cart), checkout.html, confirmation.html, and admin.html (admin dashboard). Each page is a standalone HTML file served directly by the Express.js backend.
Frontend CSS3 (Cascading Style Sheets)
CSS3 handles all visual styling of the application — colors, fonts, spacing, layout, animations, and responsive design. The project uses two CSS files: style.css for the public-facing website and admin.css for the admin panel. CSS variables (custom properties) are used for consistent theming (primary green color, accent gold). Flexbox and CSS Grid are used for responsive layouts. Media queries ensure the site works correctly on mobile phones, tablets, and desktop screens.
Frontend Vanilla JavaScript (ES6+)
Pure JavaScript (no framework like React or Vue) handles all client-side interactivity and communication with the backend API. The project is organized into four JS files:

• storage.js — Contains all async functions that call the REST API using the browser's built-in fetch() method (e.g., getProperties(), placeOrder(), userLogin()).
• main.js — Renders shared UI components like the header, footer, and property cards dynamically on every page.
• cart.js — Manages the shopping cart using localStorage to persist cart items across page navigation.
• admin.js — Powers the entire admin panel: loading data into tables, handling forms, rendering analytics charts, and managing profile updates.

2. Backend Technologies

Backend Node.js
Node.js is a JavaScript runtime environment that allows JavaScript to run on the server side (outside the browser). It is built on Google Chrome's V8 engine and uses a non-blocking, event-driven architecture, making it efficient for handling multiple simultaneous requests. In this project, Node.js executes server.js which starts the web server, connects to the database, and processes all incoming HTTP requests from the browser. The project uses Node.js version 20 (specified in .node-version) for compatibility with all dependencies.
Backend Express.js
Express.js is a minimal and flexible web application framework for Node.js. It simplifies the process of creating a web server and defining API routes. In Kigali Estates, Express.js is used to define all 25+ REST API endpoints such as GET /api/properties, POST /api/orders, POST /api/login, and DELETE /api/users/:id. It also serves all static HTML, CSS, and JavaScript files to the browser using express.static(). The body parser middleware (express.json) is configured with a 5MB limit to support base64 image uploads for property photos and admin profile pictures.

3. Database

Database NeDB (@seald-io/nedb)
NeDB is a lightweight, embedded, pure JavaScript database that stores data in flat .db files on the server's filesystem. It uses a MongoDB-like API with async methods such as findAsync(), insertAsync(), updateAsync(), and removeAsync(). Each collection is a separate file stored in the data/ folder:

• properties.db — All property listings
• agents.db — Real estate agents
• inquiries.db — Customer inquiries and order notifications
• orders.db — Customer orders from checkout
• users.db — Registered customer accounts
• settings.db — Company information and site settings
• admin.db — Admin account with hashed password and profile photo

NeDB was chosen over SQLite (better-sqlite3) because SQLite requires native C++ compilation which failed on Node.js v24. NeDB is 100% pure JavaScript and works on any Node version without any build tools.

4. Security & Authentication

Security bcryptjs
bcryptjs is a pure JavaScript library for hashing passwords using the bcrypt algorithm. When a user registers or an admin updates their password, the plain-text password is passed through bcrypt.hash(password, 10) — the number 10 is the "salt rounds" which determines how computationally expensive the hash is (higher = more secure but slower). The resulting hash (a long random string) is what gets stored in the database — the actual password is never saved. On login, bcrypt.compare(enteredPassword, storedHash) is used to verify the password without ever decrypting the hash. This protects user passwords even if the database is compromised.
Security sessionStorage (Browser API)
sessionStorage is a built-in browser Web Storage API used to maintain the logged-in user's session on the client side. After a successful login, the user's name, email, and role are stored in sessionStorage as a JSON string. This data is read on every page to show the user's name in the header and to protect admin routes. Unlike localStorage, sessionStorage is automatically cleared when the browser tab is closed, which prevents unauthorized access from shared computers.

5. DevOps & Deployment

DevOps Docker
Docker is a containerization platform that packages an application and all its dependencies into a single portable unit called a container. The Dockerfile in this project uses node:20-alpine as the base image (a lightweight Linux distribution), copies the application code, installs production dependencies with npm install --production, and starts the server with node server.js. The docker-compose.yml file defines the service configuration including port mapping (3000:3000) and a named volume (db_data) that mounts to /app/data inside the container, ensuring the NeDB database files persist even when the container is restarted or rebuilt.
DevOps GitHub & GitHub Actions (CI/CD)
GitHub is the version control platform hosting the project source code. Every change is committed with a descriptive message, maintaining a full history of the project's development (18+ commits).

GitHub Actions is the CI/CD (Continuous Integration / Continuous Deployment) platform built into GitHub. The pipeline is defined in .github/workflows/deploy.yml and triggers automatically on every push to the main branch. It runs the following steps:

1. Checkout code — downloads the latest code
2. Set up Node.js 20 — installs the correct runtime
3. Install dependencies — runs npm install
4. Run tests — runs npm test
5. Build Docker image — builds and verifies the container
6. Deploy to Render — sends a POST request to the Render deploy hook URL (stored as a GitHub secret) to trigger a live deployment
Deployment Render
Render is a cloud hosting platform used to deploy and serve the Kigali Estates application publicly on the internet. The Node.js web service runs on Render's infrastructure with the start command node server.js. A persistent disk (1GB) is mounted at /app/data via render.yaml configuration, ensuring the NeDB database files are not wiped on each deployment. The application is accessible at https://kigali-estates.onrender.com. Deployments are triggered automatically by the GitHub Actions CI/CD pipeline via a secure deploy hook URL.
